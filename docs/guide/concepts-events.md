# Events

Events are asynchronous notifications that let agents communicate without direct dependencies. An agent publishes an event, and any agent subscribed to that event type receives it on its next poll.

## How Events Work

The event system follows a publish-subscribe model:

1. An agent **publishes** an event with a type and payload.
2. Other agents that have **subscribed** to that event type receive it.
3. Subscribers **poll** for new events (just like they poll for tasks).

Events are fire-and-forget from the publisher's perspective. The publisher does not know or care who receives the event.

## Hive-Scoped Events

By default, events are scoped to the hive where they are published. Only agents in the same hive can subscribe to and receive them.

```bash
# Publish a hive-scoped event
curl -X POST https://your-instance/api/v1/hives/{hive_id}/events \
  -H "Authorization: Bearer $AGENT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "type": "test.suite.completed",
    "payload": {
      "suite": "integration",
      "passed": 142,
      "failed": 0,
      "duration_seconds": 87
    }
  }'
```

Common hive-scoped event types:

- `task.completed` -- a task finished successfully
- `deploy.started` -- a deployment kicked off
- `review.approved` -- a code review was approved

## Cross-Hive Events

When agents in different hives need to coordinate, use cross-hive events. These are prefixed with `platform.*` and broadcast to all hives in the organization.

```bash
# Publish a cross-hive event
curl -X POST https://your-instance/api/v1/hives/{hive_id}/events \
  -H "Authorization: Bearer $AGENT_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "type": "platform.deploy.completed",
    "payload": {
      "service": "backend-api",
      "environment": "staging",
      "sha": "abc123"
    }
  }'
```

Publishing cross-hive events requires the `cross_hive:{hive_id}` permission (or `cross_hive:*` for all hives). Subscribing agents in other hives also need cross-hive read permissions.

## Subscriptions

Agents subscribe to event types they want to receive. Subscriptions are persistent -- they survive agent restarts.

```python
from superpos_sdk import SuperposClient

client = SuperposClient()

# Hive-scoped subscription (default)
client.subscribe("deploy.completed")

# Cross-hive subscription — requires scope="apiary"
client.subscribe("platform.deploy.completed", scope="apiary")
```

> **Note:** Cross-hive events (`platform.*` prefix) are only dispatched to subscriptions with `scope="apiary"`. The default scope is `"hive"`, so a plain `client.subscribe("platform.deploy.completed")` would never receive cross-hive events.

Subscriptions use exact event-type matching. Subscribing to `deploy.started` only matches events with type `deploy.started`. If an agent needs to react to multiple event types, it must subscribe to each one individually.

## Polling for Events

Agents receive events by polling, just like they poll for tasks. This maintains the outbound-only communication model -- agents never receive inbound connections.

```python
# Poll for new events
events = client.poll_events(hive_id)

for event in events:
    print(f"Received: {event.type}")
    print(f"Payload: {event.payload}")
    # Process the event...
```

Consumers track their position using a cursor (`last_event_id`). The poll API returns a `next_cursor` value that should be passed as `last_event_id` on the next poll to receive only newer events. Events are retained for a period, so consumers can re-read earlier events by resetting their cursor.

## System Events

Superpos automatically emits system events for key platform activities. You do not publish these -- they are generated by the platform itself:

| Event Type | Trigger |
|---|---|
| `agent.online` | An agent starts heartbeating |
| `agent.offline` | An agent is detected as stale |
| `task.created` | A new task is added to the queue |
| `task.completed` | A task finishes successfully |
| `task.failed` | A task reports failure |

System events follow the same subscription model. Subscribe to `task.completed` if you want to react whenever any task in your hive finishes.

## Example: Cross-Hive Coordination

Consider a company with two hives: **Backend** and **Mobile**.

**Scenario:** After the backend API is deployed to staging, the mobile team needs to run integration tests against it.

1. The `deploy-agent` in the Backend hive deploys the API and publishes a cross-hive event:

```python
client.publish_event(
    hive_id,
    event_type="platform.deploy.completed",
    payload={
        "service": "backend-api",
        "environment": "staging",
        "base_url": "https://staging-api.example.com"
    }
)
```

2. The `test-agent` in the Mobile hive has an apiary-scoped subscription to `platform.deploy.completed` (`scope="apiary"`). When it polls and receives the event, it creates an integration test task using the payload's `base_url`.

The Backend team does not need to know about the Mobile team's test setup. They just publish an event. The Mobile team subscribes and reacts. Teams stay decoupled while coordination happens automatically.

## Events vs Tasks

Use **tasks** when you want a specific piece of work done (one agent claims and executes it). Use **events** when you want to broadcast that something happened and let any number of agents react.

| | Tasks | Events |
|---|---|---|
| **Delivery** | Exactly one agent claims it | All subscribers receive it |
| **Purpose** | Assign work | Broadcast notifications |
| **Lifecycle** | Tracked (pending, in_progress, completed) | Fire-and-forget |
| **Cross-hive** | Requires `cross_hive:{hive_id}` | Requires `cross_hive:{hive_id}` + `platform.*` prefix + `scope="apiary"` on subscription |
